# Chrome Music Lab 深度调研与本地仿站设计文档

调研日期：2026-05-21  
目标站点：https://musiclab.chromeexperiments.com/

## 1. 结论摘要

Chrome Music Lab 是 Google Creative Lab 推出的网页音乐实验集合。它的核心价值不是“专业音乐制作”，而是把音乐概念变成可以马上触摸、点击、拖拽、录音、播放的互动体验。官网自己的定义是：让学习音乐通过有趣、动手的实验变得更容易；教师也常用它连接音乐与科学、数学、艺术、舞蹈、真实乐器等课堂活动。

站点的设计原则非常清晰：

- 零门槛：无需登录、无需安装、打开实验即可玩。
- 强视觉隐喻：每个实验都把抽象概念变成一个直观物体，例如网格、圆盘、波形、琴键、弦线、彩色频谱、动物打击乐。
- 少文字：主界面几乎只保留标题、返回、帮助、播放/暂停等必要控件。
- 即时反馈：点击、拖拽、绘画、录音后马上得到声音和动画反馈。
- 教育优先：每个实验只讲一个概念，避免专业软件式复杂菜单。

本地仿站的可行方向：不要尝试一比一复制原站全部源码和联网协作能力，而是创建一个“体验等价”的本地版本。首页、视觉系统、实验入口、主要交互、声音反馈、基础动画应尽量接近；高级能力如 Shared Piano 的多人实时房间、Song Maker 的云端分享链接、麦克风录音可作为后续增强。

## 3. 站点整体定位

Chrome Music Lab 更像“音乐科学游乐场”，不是 DAW、不是乐理课程、不是乐器模拟器。它的产品定位可以拆成三层：

- 探索层：让用户通过直接操作发现“声音会怎样变化”。
- 创作层：让用户用网格、图形、琴键快速做出一段旋律、节奏或声音。
- 解释层：通过帮助按钮和视觉模型暗示背后的音乐/物理概念。

目标用户包括儿童、音乐初学者、教师、家长和对声音可视化感兴趣的普通用户。站点不要求用户先理解音高、频率、泛音、和弦、琶音等术语，而是让用户先玩，再感知概念。

## 4. 首页信息架构与视觉设计

首页是一个极简实验索引。

结构：

- 顶部固定白色导航条。
- 左侧品牌文字 `CHROME MUSIC LAB`，黑色大写，字距较大。
- 右侧只有 `Experiments` 和 `About` 两个链接。
- 主体是 4 列卡片网格，展示所有实验缩略图。
- About 区域是问答文本，没有复杂营销内容。
- 底部是 Google 标识、Privacy、Terms 和一个隐藏式小钢琴彩蛋。

视觉语言：

- 背景几乎全白，重点交给彩色实验缩略图。
- 网格卡片之间用极细灰线分隔。
- 缩略图以大面积图形插画为主，不依赖照片。
- 色彩高饱和，常见颜色包括黄色、红色、青绿、蓝紫、橙色。
- 字体感觉接近 Google 系产品的几何无衬线，标题字距明显，整体轻量、清爽。

交互特点：

- 首页卡片 hover/click 进入实验，卡片本身不承载复杂说明。
- About 是同页锚点滚动，而不是独立重页面。
- 移动端预期折叠菜单和网格列数自适应。

本地仿站应优先复刻这个骨架：白底、顶部导航、4 列实验图块、点击进入实验、About 问答、底部简洁信息。

## 5. 全站通用实验页设计

大多数实验页共享一个非常克制的框架：

- 顶部白色栏。
- 左侧返回箭头。
- 中间大写实验标题。
- 右侧黄色圆形 `?` 帮助按钮。
- 主体区域几乎全部交给实验画布。

这种设计把“站点外壳”压到最低，让用户注意力集中在交互对象上。实验页的交互通常是全屏画布、网格、琴键或单个大型控制器，而不是表单/卡片/菜单。

本地仿站应抽象出统一的 `ExperimentShell`：

- `Back` 返回首页。
- `Title` 居中。
- `Help` 打开说明弹窗。
- `Canvas/Playground` 填满剩余空间。
- 每个实验只在底部或侧边保留必要控制。

## 6. 14 个实验拆解

### 6.1 Shared Piano

定位：多人共享钢琴房间。

原站体验：

- 打开后生成房间链接。
- 中心是宽幅钢琴键盘。
- 支持多人实时协作。
- 支持切换乐器，例如 Drum Kit、Marimba、Strings、Woodwind、Piano、Synth。
- 有 Live、房间号、复制链接、清除、设置等控件。

设计重点：

- 视觉上像一个在线房间，而不是单机练习琴。
- 琴键横向铺满，底部状态栏承载协作信息。
- 声音反馈必须非常即时。

本地 MVP：

- 单机钢琴键盘。
- 支持鼠标点击和电脑键盘触发音符。
- 乐器切换用不同波形/音色模拟。
- 房间与多人协作先做成静态 UI，不接 WebSocket。

### 6.2 Song Maker

定位：最核心的创作工具，用网格制作并分享歌曲。

原站体验：

- 主体是音高 x 时间的网格。
- 用户点击方格添加/删除旋律音符。
- 底部有打击乐节奏轨。
- 有播放、重启、撤销、保存、设置、节拍速度、乐器选择、鼓音色选择、麦克风等控件。
- DOM 里可见关键控件：Play、Marimba、Electronic、Mic、Settings、Undo、Save、Tempo。

设计重点：

- 网格把“时间从左到右、音高从低到高”直观呈现。
- 彩色块表示音符，不用传统五线谱。
- 节奏轨与旋律轨在同一时间轴上。

本地 MVP：

- 16 列 x 8 行旋律网格。
- 2 行鼓点轨。
- 播放指针从左到右循环。
- 点击格子增删音符。
- Tempo 滑杆。
- 2-3 种音色切换。
- 保存先做本地 JSON / URL hash，不做云端分享。

### 6.3 Rhythm

定位：节奏/鼓机实验。

原站体验：

- 使用可爱的动物/角色打击乐器。
- 用户在节拍格上放置鼓点。
- 循环播放时角色按节拍敲击。

设计重点：

- 把“节奏循环”拟人化，让儿童更容易理解。
- 声音和角色动作同步是体验核心。

本地 MVP：

- 3 个打击乐角色。
- 8 或 16 步节奏格。
- 点击格子开关鼓点。
- 播放时高亮当前拍，角色产生敲击动画。

### 6.4 Spectrogram

定位：声音频谱可视化。

原站体验：

- 展示声音从低频到高频的彩色频谱。
- 可使用不同声音样本或麦克风输入。
- 视觉类似彩虹热力图，横向表示时间，纵向表示频率。

设计重点：

- 不是让用户编辑声音，而是“看见声音”。
- 黑/深色背景和彩色频谱形成强对比。

本地 MVP：

- 使用 Web Audio AnalyserNode 分析内置振荡器或音频样本。
- Canvas 绘制滚动频谱。
- 提供几个按钮：人声、长笛、鼓、麦克风。
- 麦克风权限可后置；先保证内置样本可视化。

### 6.5 Sound Waves

定位：声波传播模型。

原站体验：

- 点击键盘发声。
- 蓝色点阵以波动形式运动，模拟空气分子振动和声波传播。

设计重点：

- 用点阵运动讲物理概念。
- 键盘是触发器，点阵是解释器。

本地 MVP：

- 底部小键盘。
- 中部蓝色点阵。
- 点击音符时生成从左到右传播的波纹。
- 音高影响波纹密度或速度。

### 6.6 Arpeggios

定位：琶音与和弦分解。

原站体验：

- 彩色圆盘按音名分区。
- 中央播放按钮。
- 点击不同扇区选择和弦根音。
- 音符按一定模式逐个播放。

设计重点：

- 圆形结构暗示循环、和声关系和节奏模式。
- 字母音名直接贴在色块上。

本地 MVP：

- 12 色圆盘。
- 根音选择。
- Major/Minor 切换。
- Up、Down、UpDown 等琶音模式。
- 播放时扇区或音符点高亮。

### 6.7 Kandinsky

定位：绘画转声音。

原站体验：

- 空白画布。
- 用户画线、圆、三角或涂鸦。
- 图形被映射为不同声音。
- 灵感来自 Kandinsky 将绘画与音乐联系起来的观念。

设计重点：

- 画布必须轻、空、自由。
- 形状识别可以粗略，但声音反馈要让图形差异可感知。

本地 MVP：

- Canvas 自由绘画。
- 简单识别点击/拖拽路径：短点、长线、闭合形状。
- 播放时按 x 轴顺序触发声音。
- 提供清除、播放。

### 6.8 Voice Spinner

定位：录音、旋转、倒放与速度变化。

原站体验：

- 中心是唱片/圆形波形。
- 用户录音后通过旋转控制播放方向和速度。
- 可以慢、快、正放、反放。

设计重点：

- “用手转唱片”是交互隐喻。
- 录音属于强个人化输入。

本地 MVP：

- 初版用预置音频/合成语音波形模拟。
- 圆盘拖拽控制速度和方向。
- 后续加入 MediaRecorder 录音。

### 6.9 Harmonics

定位：泛音列。

原站体验：

- 显示多个波形/弦段。
- 用户触发不同泛音，听到自然倍频关系。

设计重点：

- 用长度、频率、波形同时解释“一个基音如何产生多个泛音”。
- 数学关系要视觉化。

本地 MVP：

- 8 条横向弦/波形。
- 点击第 n 条播放基频 n 倍的声音。
- 波形动画随频率变化。

### 6.10 Piano Roll

定位：自动钢琴卷帘/可视化乐曲。

原站体验：

- 展示预置古典乐曲的音符块滚动。
- 可切换声音和播放速度。
- 帮助用户理解多声部、音长、时间轴。

设计重点：

- 像“看一首曲子在机器里跑”。
- 音符块越长，持续时间越长；位置越高，音高越高。

本地 MVP：

- 内置 2-3 段简短旋律数据。
- 音符块从右向左或上向下滚动。
- 播放线固定，音符经过时发声。
- 支持 Piano/Synth 音色。

### 6.11 Oscillators

定位：基础波形与频率。

原站体验：

- 通过拖动改变振荡器频率。
- 展示不同波形：sine、square、triangle、sawtooth。
- 视觉上通常有拟人/弹性动画。

设计重点：

- 用户不需要懂合成器，只要感到“拖高，音变高；换形状，音色变了”。

本地 MVP：

- 4 个波形角色/卡片。
- 拖动或上下移动改变频率。
- Web Audio OscillatorNode 实时发声。
- 角色随频率缩放/抖动。

### 6.12 Strings

定位：弦长与音高关系。

原站体验：

- 多根不同长度的弦。
- 点击/拨动弦发声。
- 弦越短，音越高。

设计重点：

- 物理关系直接可见。
- 动画应像弦被拨动后的振动。

本地 MVP：

- 多条不同长度横线。
- 点击播放对应频率。
- SVG/Canvas 绘制振动波。

### 6.13 Melody Maker

定位：简化版旋律网格。

原站体验：

- 比 Song Maker 更小、更直接。
- 用户在网格中放置单声部旋律。
- 可调速度并循环播放。

设计重点：

- 去掉复杂设置，只保留旋律创作的第一步。

本地 MVP：

- 8 或 16 步 x 8 音阶网格。
- 点击单元格创建旋律。
- 每列最多一个音。
- Play 和 Tempo。

### 6.14 Chords

定位：和弦听辨/触发。

原站体验：

- 简单键盘。
- 点击一个根音即可播放对应大三和弦或小三和弦。

设计重点：

- 把“和弦是多个音一起响”变成直接体验。
- UI 应比 Shared Piano 更简单。

本地 MVP：

- 一排琴键。
- Major/Minor 切换。
- 点击根音，同时点亮根音、三度、五度。
- 播放对应三音和弦。

## 7. 可复刻体验分级

第一阶段必须做到：

- 首页 14 个实验入口。
- 统一实验页外壳。
- 所有实验都有可见、可点击、可听的核心体验。
- Web Audio 基础发声。
- 响应式布局：桌面、平板、手机都能使用。

第二阶段再做：

- Song Maker 保存/分享 URL。
- Voice Spinner 真实录音。
- Spectrogram 麦克风输入。
- Shared Piano 多人实时协作。
- 更接近原站的插画资产与动画细节。

暂不建议第一阶段做：

- 直接搬运完整官方源码。
- 复制 Google 品牌资产作为生产级发布。
- 完整云端房间、账户或分享服务。
- 专业音频采样库。

原因：用户目标是“本地复刻体验”，不是重建 Google 的托管服务。先把交互模型、视觉骨架和声音反馈做对，体验收益最高。

## 8. 推荐实现方案

### 方案 A：纯静态 HTML/CSS/JS

优点：依赖少、可直接打开、适合快速复刻。  
缺点：14 个实验状态和组件复用会变乱，后续维护成本高。

### 方案 B：React + Vite 单页应用

优点：适合拆分 14 个实验组件；状态管理清晰；适合做路由、共享组件、Canvas/Web Audio hooks。  
缺点：需要构建工具；初始文件比纯静态多。

### 方案 C：直接尝试运行官方仓库

优点：理论上最接近原站部分实验。  
缺点：官方仓库只开放部分实验、已归档、技术栈老旧，且不包含当前首页全部体验；本地整合成本不可控。

推荐：方案 B。用 React + Vite 做本地仿站，视觉和交互手写实现，必要时参考官方开源仓库的技术思路，而不是依赖旧源码运行。

## 9. 本地仿站设计

页面结构：

- `/`：首页，14 个实验图块 + About。
- `/#/experiment/song-maker` 等：实验页。
- 每个实验使用统一 shell，不额外创建多套布局。

核心模块：

- `AudioEngine`：封装 AudioContext、播放单音、和弦、鼓点、噪声、振荡器。
- `ExperimentShell`：返回、标题、帮助弹窗、主舞台。
- `HomeGrid`：首页实验入口。
- `useSequencer`：节拍器、当前步进、循环播放。
- `CanvasSurface`：绘图、频谱、波形、点阵等实验共享画布基础。

视觉系统：

- 背景：白色。
- 分割线：浅灰 `#e6e6e6`。
- 文本：近黑 `#222`。
- 品牌标题：大写、粗体、字距 0.08em 到 0.14em。
- 实验色：使用高饱和色块，例如 `#f44336`、`#ffb300`、`#8bc34a`、`#26a69a`、`#42a5f5`、`#5c6bc0`、`#ab47bc`。
- 交互反馈：播放列高亮、点击缩放、形状弹性、Canvas 动画。

声音系统：

- 初版使用 Web Audio 原生 OscillatorNode、GainNode、BiquadFilterNode、AnalyserNode。
- 鼓点可用短噪声包络和低频振荡器模拟。
- 不引入大型音频采样，避免依赖和加载成本。

## 10. 验收标准

视觉验收：

- 首页第一屏能立刻让人识别出 Chrome Music Lab 式的白底 + 彩色实验网格。
- 实验页顶部栏与原站结构一致：返回、居中标题、帮助。
- 14 个实验的图块都不是普通卡片列表，而是有对应视觉隐喻。
- 手机端不横向溢出。

交互验收：

- 每个实验至少有一个核心交互会改变画面并产生声音。
- Song Maker、Melody Maker、Rhythm、Arpeggios、Piano Roll 必须能循环播放。
- Oscillators、Strings、Harmonics 必须能体现频率/音高变化。
- Spectrogram 必须有实时可视化。
- Kandinsky 必须能绘画并回放。
- Chords 必须能同时播放多个音。

技术验收：

- 本地 `npm run dev` 可启动。
- `npm run build` 通过。
- 无远程运行时依赖，除非明确标注为可替换资源。
- Web Audio 初始化遵守浏览器策略：第一次用户点击后启动 AudioContext。

## 11. 实施顺序

建议按体验价值排序，而不是按首页顺序：

1. 搭建 React + Vite、首页网格、统一实验 shell。
2. 建立 AudioEngine 和基础音符/频率映射。
3. 实现 Song Maker，因为它是站点最核心体验。
4. 实现 Melody Maker、Rhythm、Chords、Arpeggios，复用 sequencer。
5. 实现 Oscillators、Strings、Harmonics，复用 oscillator/频率模型。
6. 实现 Sound Waves、Spectrogram、Kandinsky，补 Canvas 可视化。
7. 实现 Piano Roll、Voice Spinner、Shared Piano 的本地版。
8. 做视觉统一、响应式、浏览器验证和体验修正。

## 12. 关键风险

- 声音自动播放限制：浏览器要求用户手势后才能启动 AudioContext，需要设计“首次点击启用声音”。
- 麦克风权限：Voice Spinner 和 Spectrogram 的真实输入可能被权限或非 HTTPS 限制影响，本地先做降级。
- 多人协作：Shared Piano 的在线房间不适合作为第一阶段目标。
- 一比一视觉资产：原站有大量插画与 Canvas/WebGL 细节，本地仿站应优先“体验等价”，再逐步提高素材还原度。
- 商标与发布边界：本地学习仿站可以复刻体验，但如果公开发布，应避免冒充 Google 或直接使用受保护品牌资产。

## 13. 本次实现范围建议

第一版目标：做一个可运行的本地 Chrome Music Lab 风格仿站，包含首页和 14 个实验的本地可玩 MVP。它应让用户感到“这是同一种体验逻辑”：进入实验、直接玩、声音与画面同步、少文字、强颜色、强隐喻。

不把第一版目标定义为“完全复制每个实验的原始算法、联网能力和素材”。这会显著拉长周期，并且和本地体验目标不完全一致。
